# goal-prompt-builder

ユーザーの開発指示を受け取り、Claude Codeの`/goal`に渡す**プロンプト**を設計してファイルに書き出すスキル。コードは書かず、プロンプト設計に集中する。

<core_principle>
`/goal`は、**独立した検証役が完了条件を満たしたと判定するまで**、調査→実装→検証を自律的に繰り返す。だからプロンプトの成否は**完了条件の質**でほぼ決まる。

- 曖昧 → 検証役が判定できず止まらない（または的外れな状態で誤完了する）
- 外形的に検証できない → 実装役が「たぶん直った」で止まり、実際は動かない
- 抜け穴あり → テスト削除やハードコードで条件だけ満たし、中身が壊れる（ループはミスをスケールさせる）

**このスキルの仕事の大半は、誰が検証しても同じ判定になる、機械的に確認できる完了条件を確定すること。** テンプレートの穴埋め作業ではない。
</core_principle>

<workflow>

### 1. 指示を理解する

ゴールを一文で言い切れるまで読み解く。複数の目的が混在するなら、分割か主目的への絞り込みを提案する（1回の`/goal`は単一ゴールほど止まりやすい）。対象範囲はコードベースを調べて把握する。

### 2. 検証可能な完了条件を確定する（最重要）

ループの心臓部。手を抜くと後段すべてが崩れる。

- **検証手段を実物から拾う**: テスト・lint・型チェック・ビルドの実行コマンドや再現手順を`package.json`/`Makefile`/`Taskfile`/CI設定/既存テストから特定する。<critical>存在を確認していないコマンドを推測で書かない。検証役が永遠に判定できなくなる。</critical>
- **曖昧なら必ず質問する**: 「どうなれば完了か」「どのコマンド・挙動で検証するか」が一意に定まらないなら、`AskUserQuestion`で確認してから進む。推測で埋めると、的外れな条件でループが何時間も空回りする。条件の質が最優先で、速さは二の次。
- **粒度を上げる**: 「テストが通る」ではなく「`<コマンド>`がexit 0」「`/login`へのPOSTが401でなく200を返す」のように、第三者が見ても判定がブレない形にする。

### 3. プロンプトを組み立てる

下記`<template>`に沿って日本語で生成する。プレースホルダは具体値で埋め切る（`<...>`を残さない）。

### 4. ファイルに書き出して報告

`goal-prompts/<YYYYMMDD>-<短いスラッグ>.md`に書き出し（ディレクトリがなければ作成）、パスを提示して「内容を`/goal`に渡してループを開始してください」と添える。推測で埋めた箇所やユーザー確認が要る点があれば明示する。

</workflow>

<example>

**入力（ユーザーの指示）:**
> ログインフォームで、メールアドレスが空のまま送信するとサーバーエラー500になる不具合を見つけたので修正して

**生成されるプロンプト**（コードベース調査で`npm test`と該当ファイルを特定済みと仮定）:

```markdown
# ゴール

ログインフォームでメールアドレスが空のまま送信されたとき、500エラーではなく適切なバリデーションエラー（400）を返すよう修正する。

# 背景・前提

- 該当エンドポイント: `src/api/auth/login.ts`の`POST /api/login`ハンドラ
- 現状: `email`が空文字のとき`findUserByEmail()`がnullを返し、続く処理のnull参照で500になる
- バリデーションは`src/api/auth/validators.ts`に既存パターンあり（パスワード長チェック等）。同じ流儀で追加する
- テスト: `npm test -- auth`で認証系テストを実行できる

# 進め方（ループ）

1. 調査: `login.ts`と`validators.ts`を読み、既存バリデーションの流儀を確認する
2. 実装: 空メールアドレスを既存パターンに沿って弾き、400を返すよう最小変更する
3. 検証: 下記「完了条件」をコマンドで確認する
4. 未達なら、失敗の原因を特定して2へ戻る

進捗の記録: 各反復の試行内容と結果を`goal-progress.md`に追記する。

# 完了条件（すべて満たしたら停止）

- [ ] 空メールアドレスのPOSTに対し400とバリデーションエラーメッセージが返る — 検証: 新規テストを追加し`npm test -- auth`がexit 0
- [ ] 正常系（有効なメール+パスワード）のログインが従来どおり成功する — 検証: `npm test -- auth`の既存ケースがexit 0
- [ ] 既存のテスト・lintを壊していない — 検証: `npm test`と`npm run lint`がexit 0

# 制約・ガードレール

- 完了条件を満たすためにテスト・アサーションを削除・弱体化・スキップしない。
- スコープ外のファイルを変更しない: 変更は`src/api/auth/`配下とそのテストに限定する。
- バリデーションは新設せず、`validators.ts`の既存パターンに合わせる。

# エスカレーション

- 3回試しても完了条件を満たせない、または原因が`login.ts`外にあると判明した場合は、試したことと現状を`goal-progress.md`にまとめて停止し、人間に判断を仰ぐ。
```

</example>

<constraints>
IMPORTANT: 以下を必ず守ること。

- このスキルはプロンプトを設計・出力するだけ。指示された開発作業そのもの（コード修正・テスト実行）は行わない。
- 完了条件に書くコマンドは、コードベースで実在を確認したものだけにする。推測したコマンドを混ぜない。
- 完了条件が指示から一意に定まらないときは、`AskUserQuestion`で確認してから生成する。勝手に決めない。
- ユーザーへの質問は必ず`AskUserQuestion`を使うこと。
</constraints>
