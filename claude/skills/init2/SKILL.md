# init2 — CLAUDE.md ジェネレーター

CLAUDE.md は Claude Code がプロジェクトを理解するための「取扱説明書」。適切に書かれた CLAUDE.md は、毎回同じ説明をする手間を省き、Claude が意図通りに動く確率を大幅に上げる。

## フェーズ1: プロジェクト分析

まずプロジェクトのコンテキストを把握する。以下を調べる（ファイルを読む前に推測しない）：

1. `ls`, `find`, `git log --oneline -10` でプロジェクト構造を確認
2. `package.json` / `go.mod` / `Cargo.toml` / `pyproject.toml` 等でスタック特定
3. 既存の `CLAUDE.md` や `README.md` があれば読む
4. `.github/workflows/` があればCI/CDパイプラインを把握

不明点はユーザーに聞く。特に以下は必ず確認する：
- このプロジェクトで Claude に何をしてほしいか（コーディング支援 / コードレビュー / リファクタリング など）
- 特に守ってほしいルールや禁止事項はあるか

## フェーズ2: CLAUDE.md 生成

分析結果をもとに CLAUDE.md を生成する。以下のテンプレートをベースに、プロジェクトに不要なセクションは省略してよい。

```markdown

- 言語: [例: Go 1.22, TypeScript 5.x]
- フレームワーク: [例: Echo, React]
- インフラ: [例: AWS, Docker]
- テスト: [例: go test, Jest]

## 開発コマンド

[よく使うコマンドをリスト化。Claudeが毎回調べずに済む]

\`\`\`bash
# ビルド
make build

# テスト実行
make test

# リント
make lint
\`\`\`

## コーディングルール

[プロジェクト固有のルールを明示する。なぜそうするかの理由も添える]

- エラーハンドリングは必ず行う（サイレントな失敗は本番障害の原因になるため）
- テストなしのコードはマージしない
- [その他プロジェクト固有ルール]

## 実装時の注意事項

[Claudeに特に注意してほしいことを記載]

依頼された内容のみ実装する。以下は明示的に求められない限り行わない：
- 依頼外のリファクタリングや「改善」
- コメント・docstring の追加（変更していないコードへ）
- 不要な抽象化やヘルパー関数の作成

ファイルを読む前にコードについて推測しない。必ず該当ファイルを開いてから回答する。

## ツール使用方針

[Claudeのツール使用をどう制御するか]

独立した操作は並列実行してよい（複数ファイルの読み込み、並行検索など）。
ただし以下は必ずユーザーに確認してから実行する：
- `git push --force` / `git reset --hard`
- ファイルの削除
- 本番環境への変更
- 外部サービスへのメッセージ送信

## ディレクトリ構成

\`\`\`
[主要ディレクトリの説明。Claudeがどこに何があるか分かるように]
\`\`\`
```

## セクション別ガイドライン

### 役割定義が重要な理由
役割を明示することで Claude の応答トーンと視点が定まる。「シニアエンジニア」と「ジュニアエンジニア」では同じ質問への答え方が変わる。

### 指示の書き方原則

**良い例（なぜかを添える）:**
```
ellipsis（...）を使わない。
理由: このプロジェクトの出力は音声読み上げエンジンに渡されるため、
ellipsis の読み方が未定義で不自然な読み上げになる。
```

**避けるべき例:**
```
NEVER use ellipses
```

後者も機能するが、前者の方が Claude が文脈を理解して例外ケースを適切に判断できる。

### 出力フォーマットの指定

「〜するな」より「〜せよ」の形式で書く：

| 避ける | 推奨 |
|--------|------|
| マークダウンを使うな | 回答は plain text の段落で記述せよ |
| 長い回答はするな | 要点を3行以内で答えよ |
| コードブロックを多用するな | コードは \`\`\` ブロックのみに限定し、説明は prose で書け |

### ツール使用の制御

積極的に動いてほしい場合:
```
意図が明確なら実装まで進めてよい。
不明点はツールで調べて解決し、推測で止まらない。
```

慎重に動いてほしい場合:
```
変更前に必ず確認を取る。
ファイル編集・実行・削除は明示的な指示があってから行う。
```

### エージェント安全性

不可逆な操作（削除、force push、本番変更）は明示的にリストアップしておく。
Claude はリストにある操作を見たとき、自動的に一時停止してユーザーに確認する。

## 生成後の確認

CLAUDE.md を生成したら、ユーザーに以下を確認する：

1. 不足しているルールや文脈はないか
2. 逆に不要なセクションはないか
3. プロジェクト固有の禁止事項（触れてはいけないファイル、変えてはいけない設定など）があれば追加する
