# Green到達戦略

失敗するテスト（Red）を通すとき、いきなり完璧な実装を書く必要はない。
Kent Beckは「テストを通すための3つの戦略」を挙げている。状況に応じて使い分け、
**歩幅（一度に進む大きさ）**を調整するのが狙いである。

## 1. 仮実装 (Fake It)

期待値をそのままベタ書きで返してテストを通す。最小・最速で緑にする。

```python
# テスト: add(2, 3) == 5
def add(a, b):
    return 5   # まずベタ書きで通す
```

通った後、別のテストや三角測量で一般化していく。

**いつ使うか**: 実装の見通しに自信がないとき、まず緑の状態を手に入れて足場を作りたいとき。
緑→赤を行き来しながら少しずつ正しい実装へ寄せていける安心感が得られる。

## 2. 三角測量 (Triangulation)

複数の具体例を用意し、それらを同時に満たすように一般化を導く。
1つの例だけでは仮実装で通ってしまうので、2つ目の例を足して「ベタ書きでは通らない」
状況を作り、一般化を強制する。

```python
# test1: add(2, 3) == 5
# test2: add(10, 7) == 17   ← これを足すと return 5 では通らない
def add(a, b):
    return a + b            # 一般化が導かれる
```

**いつ使うか**: 正しい一般化の形に確信が持てず、例を増やして輪郭を浮かび上がらせたいとき。

## 3. 明白な実装 (Obvious Implementation)

正しい実装が自明なら、仮実装を経ず直接書く。

```python
def add(a, b):
    return a + b   # 自信があるなら最初からこれでよい
```

**いつ使うか**: 実装が単純で確信があるとき。歩幅を大きく取れる。
ただし「明白だと思ったのに赤になる」こともある。そのときは素直に歩幅を縮め、
仮実装や三角測量に切り替える。

## 使い分けの感覚

- 確信が高い → 明白な実装（大きい歩幅）
- 確信が低い・複雑 → 仮実装で足場を作る（小さい歩幅）
- 一般化の形が見えない → 三角測量で例を足す

重要なのは、**詰まったら歩幅を小さくする**こと。赤が続いて不安なときほど、
仮実装で一度緑に戻り、小さなステップで前進する。TDDのテンポを失わないための調整弁である。
