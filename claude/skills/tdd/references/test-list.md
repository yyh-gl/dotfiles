# テストリストの書き方

テストリストはCanon TDDの出発点であり、最も軽視されがちな部分でもある。
ここを丁寧にやるほど、後のサイクルが迷わず回る。

## テストリストとは何か

「実装が完成したら通っていてほしいテストシナリオ」を、**自然言語で箇条書き**にしたもの。

- コードではない。テストコードを書くのはStep 2から。
- 仕様の理解を整理する場でもある。何を作るのかを利用者視点で言語化する。
- 生きたメモである。実装中に気づいたケースはどんどん追記し、片付いた項目には印を付ける。

## 何を含めるか

ひとつの振る舞いについて、次の観点で漏れがないか考える。

- **正常系** — 代表的な入力に対する基本の振る舞い
- **境界値** — 空・0・1件・最小・最大・ちょうどの値など、境目になる入力
- **特殊ケース** — 重複・順序・null/未定義・型の揺れなど
- **異常系** — 不正な入力やエラー時にどう振る舞うべきか

## 粒度の指針

- 1項目 = 1つの観察可能な振る舞い。「いくつかのことをまとめて確認する」項目は分割する。
- 1項目が大きすぎてテスト1つに落とせないと感じたら、もっと小さい具体例に割る。
- 「どう実装するか」ではなく「どう振る舞うか」で書く。実装手順はリストに書かない。

## 良い例 / 悪い例（FizzBuzz）

**良い例**（振る舞い単位・具体的）:

```
- 1 を渡すと "1" を返す
- 2 を渡すと "2" を返す
- 3 を渡すと "Fizz" を返す
- 5 を渡すと "Buzz" を返す
- 15 を渡すと "FizzBuzz" を返す
- 3の倍数（6, 9, ...）で "Fizz" を返す
- 5の倍数（10, 20, ...）で "Buzz" を返す
- 3と5両方の倍数（30, 45, ...）で "FizzBuzz" を返す
```

**悪い例**（粒度が粗い・実装目線・曖昧）:

```
- FizzBuzzが正しく動く          ← 何を確認するのか不明。テストに落とせない
- ループでmから nまで出力する     ← 実装手順であって振る舞いではない
- 3でも5でも割り切れる数を処理する ← "FizzBuzz" を返す、と振る舞いで書く
```

## 実装が始まってから

実装中に「あ、このケースも要る」と気づくのは正常。手を止めてその場で実装するのではなく、
**リストに1行足して後回し**にする。目の前の1テストに集中し続けることが、小さく速く
回すコツになる。
