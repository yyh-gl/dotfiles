## 目的

Webサービスを本番公開する直前に、「見落とすと公開後に痛い目を見る」項目を体系的に潰す。
個人開発〜小規模サービスで実際に事故りやすいポイントを集めたチェックリストにもとづき、
**対象コードベースを自分で読んで監査し**、項目ごとに状態を判定したレポートを残す。

なぜレポートを残すのか: 公開前チェックは一度やって終わりではなく、修正→再点検を繰り返す。
ファイルに状態（OK/NG/要手動確認/該当なし）と根拠が残っていれば、何が片付いていて
何が宿題かを誰でも追える。口頭やチャットに流すだけだと取りこぼす。

このスキルは**機械的なチェックリスト穴埋めではない**。各項目が「なぜ重要か」を理解した上で、
そのサービスの作りに即して本当にリスクがあるかを判断する。コードから断定できないものは
無理にOK/NGを付けず、正直に「要手動確認」として人間に委ねる。

## 全体の流れ

```
スタックを把握  →  checklist.md を読む  →  カテゴリごとに監査  →  レポート生成・保存
```

## Step 1: 対象スタックを把握する

何を使って作られたサービスかを最初に押さえる。検出のしかたも判定基準も、これで変わる。

- 言語・フレームワーク（package.json / go.mod / Gemfile / requirements.txt / composer.json など）
- フロントの構成（SSR/SPA/SSG、Next.js・Rails・Laravel・Django など）
- インフラ・外部サービス（Vercel/AWS/Cloudflare、DB、オブジェクトストレージ、決済SaaS、メール送信基盤）

ここで「決済機能はあるか」「メール送信はあるか」「認証はあるか」を見極めておくと、
後段で該当しないカテゴリを素早く `➖ 該当なし` に倒せる。
ただし**カテゴリ自体は常に全部レポートに載せる**（該当なしと明記することに意味がある）。

## Step 2: チェック項目の詳細を読む

`references/checklist.md` に、全カテゴリの各項目について「何を確認するか・なぜ重要か・
どう検出するか（grepの当たり所やファイルの在り処）・自動判定できるか手動確認が要るか」を
まとめてある。**監査を始める前に必ず読む。** 量が多いので、まず目次で全体像を掴み、
カテゴリを監査するたびに該当節を見返す。

## Step 3: カテゴリごとに監査する

`references/checklist.md` の順にカテゴリを1つずつ点検する。各項目について:

1. **根拠を集める** — grep/ファイル読みで、対策が入っているか・問題がないかの証拠を探す。
   推測で済ませない。例えば「CookieにHttpOnlyが付いているか」はCookie発行箇所を実際に見る。
2. **状態を判定する** — 下記の4状態のいずれかを付ける。
3. **根拠を記録する** — NG・要手動確認には必ず理由を添える。問題箇所は `file:line` で示す
   （クリックで飛べて修正が速い）。

### 状態の定義

| 記号 | 状態 | 使うとき |
|------|------|----------|
| ✅ | OK | 対策が入っている / 問題が見つからないことをコードで確認できた |
| ❌ | NG | 明確に問題がある。`file:line` と何が危ないかを示す |
| ⚠️ | 要手動確認 | コードからは断定できない。実機確認・運用・外部設定（DNS/SaaS管理画面など）が要る |
| ➖ | 該当なし | その機能自体が存在しない（例: 決済機能がない → 決済カテゴリ全体） |

`⚠️ 要手動確認` を恐れない。SPF/DKIMのDNS設定、二要素認証の有効化、実機での表示崩れなど、
コードリポジトリの外にある事柄は構造上コードから分からない。これを無理にOKにする方が危険で、
「人間がここを見る必要がある」と明示するのがこのチェックの価値の一つ。

## Step 4: レポートを生成して保存する

下記テンプレートでMarkdownレポートを作り、ファイルに保存する。
保存先は既定で対象リポジトリ直下の `service-launch-check-report.md`
（ユーザーが場所を指定したらそれに従う）。保存後、サマリーと特に重大なNGを会話でも数行で伝える。

### レポートの構成

```markdown
# サービス公開前チェック レポート

- 対象: <サービス名 / リポジトリ>
- スタック: <言語・FW・主要インフラ>
- 実施日: <YYYY-MM-DD>

## サマリー

| カテゴリ | ✅ OK | ❌ NG | ⚠️ 要手動確認 | ➖ 該当なし |
|----------|------:|------:|------:|------:|
| セキュリティ | 8 | 2 | 5 | 1 |
| … | | | | |
| **合計** | | | | |

### 公開前に必ず対応すべきNG（重要度順）

1. ❌ <一言サマリー> — `path/to/file.ts:42`
2. …

## セキュリティ

### 認証に関わるCookieの属性
- ✅ HttpOnly属性 — `src/auth/session.ts:31` でSet-Cookie時に付与済み
- ❌ Secure属性が未設定 — `src/auth/session.ts:31`。本番HTTPSでCookieが平文経路にも乗る恐れ
- ⚠️ Domain属性 — 設定箇所が見当たらず既定値依存。意図どおりか実機で要確認

（以下、checklist.md のカテゴリ順に全項目を列挙）
```

レポートでは**NGと要手動確認を埋もれさせない**のが肝心。サマリーの「公開前に必ず対応すべきNG」に
重要度順で先頭に集約し、本文では各項目に根拠を添える。OKばかりを長々書くより、
「次に何をすべきか」が一目で分かる構成を優先する。

## 注意事項

- **断定はコードの裏付けがあるときだけ。** 「たぶん大丈夫」でOKを付けない。確認できなければ `⚠️`。
- **該当しないカテゴリも省略しない。** `➖ 該当なし` と書くこと自体が「検討済み」の証拠になる。
- **重要度を意識する。** 認証・決済・個人情報まわりのNGは、ファビコン未設定とは重みが違う。
  サマリーの並びと表現でその差を伝える。
- **スタック固有の作法を尊重する。** フレームワークが既定で対策している事柄（多くのFWはテンプレート
  出力を自動エスケープするなど）は、それを根拠にOK判定してよい。車輪の再発明を疑わない。
- **このリストは出発点。** 対象サービス特有のリスク（独自の権限モデル、外部連携など）に
  気づいたら、リストにない項目でも追記してよい。
