# サービス公開前チェックリスト 詳細

各項目について「何を確認するか / なぜ重要か / どう検出するか / 判定の傾向（自動 or 要手動）」をまとめる。
出典: catnose「個人開発のサービスを公開する前に確認したいチェックリスト」の項目を再構成したもの。

検出パターン（grep等）はあくまで当たりを付けるための入口。ヒットの有無だけで断定せず、
ヒットした箇所を実際に読んで文脈を確認する。フレームワークが既定で対策している場合は、それを根拠にOKとしてよい。

## 1. セキュリティ

最も事故が痛いカテゴリ。認証・個人情報・課金に直結するため、ここのNGは最優先で扱う。

### 認証に関わるCookieの属性

セッションCookie等が漏れる・盗まれると即なりすましにつながる。発行箇所（Set-Cookie / cookie options）を実際に読む。

- **HttpOnly**: JSからCookieを読めなくしXSS時の流出を緩和。Cookie設定オプションに `httpOnly: true` 等があるか。
- **SameSite**: `Lax` または `Strict`。CSRF緩和。`sameSite` 指定を確認。決済リダイレクト等で `None` が必要なら `Secure` 必須。
- **Secure**: HTTPS経路のみ送出。本番で `secure: true` 相当か。
- **Domain属性**: 不必要に広いドメイン（親ドメイン）に撒くとサブドメイン間で漏れる。明示設定の妥当性。
- **`__Host-` プレフィックス**: より強い固定（Secure必須・Path=/・Domain無し）。採用していれば加点。
- 検出: `Set-Cookie`, `httpOnly`, `sameSite`, `secure`, `cookie(` , フレームワークのセッション設定ファイル。
- 傾向: 自動判定しやすい。ただし本番設定が環境変数依存なら `⚠️`。

### ユーザー入力値のバリデーション

- **サーバーサイドで検証しているか**: クライアント側だけの検証は無意味。APIの入口でバリデーションがあるか。
- **URL入力のチェック**: `javascript:` 等の危険スキームを弾いているか（プロフィールリンク等）。
- **HTML出力時のエスケープ/サニタイズ**: XSS。`dangerouslySetInnerHTML` / `v-html` / `innerHTML` / `raw` / `|safe` / `html_safe` の使用箇所は要精査。テンプレートエンジンの自動エスケープに乗っていればOK。
- **SQLインジェクション**: 文字列連結でSQLを組んでいないか。プレースホルダ/ORM/パラメータバインドを使っているか。`query(`, `execute(`, 文字列内の `SELECT/INSERT` + 変数結合。
- **ユーザー指定パス（ハンドルネーム等）の検証**: 任意文字列がURLパスになる場合の形式チェック。
- **リザーブド文字列**: `admin`, `api`, `settings` 等を予約し、ユーザーが取得できないようにしているか。
- 検出: 上記キーワード群、バリデーションライブラリ（zod/yup/valibot/class-validator/dry-validation等）の利用。
- 傾向: エスケープ・SQLは自動で当たりが付く。網羅性の最終確認は `⚠️` 寄り。

### レスポンスヘッダ

- **Strict-Transport-Security (HSTS)**: HTTPS強制。
- **X-Frame-Options** (または CSPの `frame-ancestors`): クリックジャッキング対策。
- **X-Content-Type-Options: nosniff**: MIMEスニッフィング抑止。
- 検出: `helmet`, `Strict-Transport-Security`, `X-Frame-Options`, `nosniff`, `Content-Security-Policy`, next.config の headers, nginx/vercel.json 等。
- 傾向: コード/設定にあれば自動OK。CDN/リバプロ側設定なら `⚠️`。

### その他セキュリティ

- **重要操作時の直近ログイン必須化**: パスワード変更・退会・メール変更等で再認証を求めるか。
- **キャッシュ対象の選別**: 個人情報を含むレスポンスにCDN/`Cache-Control: public` を当てていないか。
- **オブジェクトストレージの非公開化**: S3バケット等が意図せず公開されていないか（コードからはACL/IaC設定で判断、無ければ `⚠️`）。
- **オープンリダイレクト**: `?redirect=` 等の遷移先を検証せず使っていないか。
- **アクセス権限チェック（更新・削除時）**: 「自分のリソースか」を確認しているか。所有者チェックなしの `update/delete` は重大。
- **WHERE句の適切性**: 更新・削除クエリに本人/対象を絞る条件があるか（全件更新事故の防止）。
- **レスポンスヘッダへのユーザー入力混入防止**: ヘッダインジェクション（改行混入）。
- **エラーメッセージの情報漏洩**: スタックトレース・SQL・内部パスを本番でユーザーに返していないか。
- **ファイルアップロード検証**: 形式・サイズ・ファイル名（パストラバーサル/上書き）を検証しているか。
- **DB定期バックアップ**: 有効か（運用設定。コード外なら `⚠️`）。
- **オブジェクトストレージのバックアップ/バージョニング**: 同上。
- **クラウドサービスの二要素認証**: 管理アカウントのMFA（運用。`⚠️`）。
- 傾向: 権限チェック・オープンリダイレクト・エラー漏洩はコードで判定可能。バックアップ・MFA・ストレージ公開設定は `⚠️` になりがち。

### ログイン

- **メールアドレス本人確認**: 確認メール（verification）を経て有効化しているか。
- **登録メールアドレスの列挙防止**: 「このメールは登録済み」等で存在を漏らしていないか。エラー文言を統一。
- **複数ログイン方法の仕様統一**: メール/OAuth等を併用時、同一メールの扱いが一貫しているか。
- **メールアドレス変更・連携アカウント変更の導線**: 後から変えられるか。
- 傾向: 確認メール有無は自動、文言設計や仕様一貫性は `⚠️` 寄り。

## 2. メール送信

メールは「届かない」「迷惑メール送信源になる」事故が起きやすい。送信処理と配信基盤の両面で見る。

- **スパム送信の防止 / 不特定多数への重複送信防止**: 送信トリガに歯止めがあるか（レート制限・宛先重複排除）。
- **SPF / DKIM / DMARC**: 送信ドメイン認証。**DNS設定のためコードからは原則分からない → `⚠️`**。送信に使うサービス（SES/SendGrid/Resend/Postmark等）は特定して報告。
- **バッチ処理での重複送信防止**: ジョブ再実行・リトライで二重送信しないか（冪等性・送信済みフラグ）。
- **メルマガ購読解除の導線**: 配信停止リンクがあるか。
- **大量配信時の List-Unsubscribe: One-Click**: ヘッダ対応しているか（Gmail等の要件）。
- 検出: メール送信ライブラリ/SaaSのSDK、`sendMail`, `mailer`, テンプレート、`List-Unsubscribe`, バッチ/cron/queue 周り。
- 傾向: 送信処理・解除リンク・重複対策はコードで判定。ドメイン認証は `⚠️`。

## 3. SEO

検索/共有からの流入を毀損しないための最低限。クロールされて困るページを止める観点も含む。

- **全ページのtitleタグ**: 各ページに固有のtitleが入るか（テンプレートで動的設定されているか）。
- **重要ページのcanonical URL**: 重複URL（クエリ違い等）を正規化しているか。
- **エラーページのステータスコード**: 400系または500系, noindexになっているか。
- **検索結果ページ等の noindex / canonical**: 無限に増えるパラメータURLをインデックスさせていないか。
- **サイト全体の noindex 確認**: 開発用の `noindex` / `robots.txt: Disallow: /` を本番で外し忘れていないか。**これは公開時の典型事故**。
- **meta description**: トップ等の主要ページに設定。
- **XMLサイトマップ**: 生成・登録しているか（`sitemap.xml`）。
- 検出: `<title>`, `next/head` / `metadata`, `canonical`, `robots`, `noindex`, `sitemap`, `meta name="description"`, `robots.txt`。
- 傾向: titleやnoindexの有無は自動。Search Console登録は `⚠️`。

## 4. OGP

SNS共有時の見栄え。抜けていると共有時にカード崩れ。主にHTML/メタ生成箇所を見る。

- **og:title** / **og:description** / **og:url** / **og:image**
- **twitter:card**（`summary_large_image` 等）
- 検出: `og:`, `property="og:`, `twitter:card`, `openGraph`（Next.js metadata）。
- 傾向: ほぼ自動判定可能。動的OG画像の実体は `⚠️` で実機確認を促してよい。

## 5. 決済機能

**決済機能が無ければカテゴリ全体を `➖ 該当なし`。** ある場合は金銭事故・会計に直結するので厳しく見る。

- **会計処理方針**: 売上計上のタイミング等が決まっているか（運用・規約。`⚠️` 寄り）。
- **決済データ不整合の検知**: Webhookと自DBの突合・整合チェックがあるか。
- **重複決済の防止**: 二重サブミット・冪等キー（idempotency key）を使っているか。
- **退会時のロジック整合**: 課金中ユーザーの退会でサブスクが宙に浮かないか。
- **サブスク自動キャンセル**: 退会/条件成立時に購読を止めるか。
- **返金・日割りの規約記載**: 利用規約に明記されているか（`⚠️`、規約文面の確認）。
- **解約導線**: ユーザーが自分で解約できるUIがあるか。
- **カード期限切れ時の対応**: 失敗時のリトライ・通知。
- **領収書の適格請求書要件**: インボイス対応が必要か（`⚠️`）。
- 検出: 決済SaaS SDK（Stripe等）、`webhook`, `idempotency`, `subscription`, `refund`, `cancel`。
- 傾向: 冪等性・Webhook処理・解約導線はコードで判定。会計方針・規約・インボイスは `⚠️`。

## 6. アクセシビリティ

最低限の機械可読性。網羅監査ではなく、明らかな抜けを拾う。

- **画像のalt属性**: 意味のある画像に `alt` があるか（装飾は空altでよい）。`<img` で alt 欠落を探す。
- **SVGアイコンの aria-label**: 操作要素のアイコンにラベルがあるか（`aria-label` / `aria-hidden`）。
- 検出: `<img`, `alt=`, `<svg`, `aria-label`, アイコンボタン。
- 傾向: 半自動。alt欠落は機械的に拾えるが「意味のある画像か」は文脈判断。

## 7. パフォーマンス

体感速度とCWV。代表的な落とし穴のみ。

- **不要モジュールの削除**: 本番バンドルにdevツールやデバッグ用が混ざっていないか（`console.log` 残し、巨大依存）。
- **静的ファイルのCDNキャッシュ**: 静的アセットに長期キャッシュ + ハッシュ付きファイル名か。
- **画像のレイアウトシフト防止**: `width`/`height` または `aspect-ratio` 指定で CLS を防いでいるか。
- **画像サイズの最適化**: 適切なフォーマット/リサイズ（next/image 等の最適化）を使っているか。
- **SQLインデックス**: 頻出クエリの絞り込み/結合キーにインデックスがあるか（マイグレーション/スキーマを確認）。
- 検出: バンドラ設定、`Cache-Control`, `<img` の寸法指定, `next/image`, マイグレーション内 `index`/`CREATE INDEX`。
- 傾向: コードで当たりが付く。実測値（Lighthouse等）は `⚠️`。

## 8. 複数環境での動作確認

**本質的に実機確認が中心 → 多くが `⚠️ 要手動確認`。** ただしコード/CSSから危うい箇所は指摘する。

- **モバイル・タブレット表示**: レスポンシブ対応があるか（メディアクエリ/レスポンシブユーティリティの有無は見える）。
- **各OS/ブラウザ対応**: Safari固有の挙動等。実機確認前提。
- **スクロールバー常時表示**: 横はみ出し・`100vw` 起因のずれ。
- **ユーザー入力が長い場合の表示崩れ**: 長い名前/URLで `overflow-wrap`/`break-word`/`truncate` 等の処置があるか。
- 検出: メディアクエリ、`overflow`, `break-word`, `truncate`, `text-overflow`。
- 傾向: 対応の「気配」はコードで見えるが、崩れの有無は実機。基本 `⚠️`、明確な抜けは `❌`。

## 9. その他

公開直後に「やっておけば」となりがちな細々した項目。

- **ローカルストレージの7日削除等の方針**: 古いデータの掃除（仕様。`⚠️` 寄り）。
- **サードパーティCookie非依存**: 廃止に備え依存していないか。
- **HTML lang属性**: `<html lang="ja">` 等が正しいか。
- **サーバーエラーの通知/検知**: エラー監視（Sentry等）が入っているか。**入っていないと公開後に障害に気づけない。重要度高め。**
- **404・50x系ページのデザイン**: 通常の動作で起こり得るエラーである＝悪い挙動とは限らないことがユーザーに伝わるデザインになっているか（次のアクションへの導線が表示されているなど）。
- **ファビコン**: 設定されているか（`favicon`）。
- **apple-touch-icon**: 設置されているか。
- **アクセス解析**: 解析ツール（GA等）を導入しているか。
- **電気通信事業者の届出**: サービス内容によって要否（法務。`⚠️`、必要性に言及するだけでよい）。
- **サービス名の多言語チェック**: 名称が他言語で不適切な意味にならないか（`⚠️`）。
- 検出: `lang=`, `Sentry`/エラー監視SDK, `favicon`, `apple-touch-icon`, 404ページ, 解析タグ。
- 傾向: lang/favicon/監視/404はコードで判定。法務・命名は `⚠️`。
