# claude-config-check — Claude Code設定の監査

対象プロジェクトのClaude Code設定（ステアリング機構）が、Anthropic公式の指針に沿って
**正しく・効率的に**組まれているかを点検し、レポートとして残す。

## 何を見るのか — 前提となる考え方

Claude Codeには指示を効かせる手段が複数あり、それぞれ「いつロードされるか」「コンパクション後も
残るか」「コンテキストコストはどれだけか」が違う。**適材適所が崩れると、トークンを浪費したり、
効かせたい指示が効かなかったりする**。このスキルは次の機構を横断的に監査する。

| 機構 | 置き場所 | 本質 |
|------|----------|------|
| CLAUDE.md | ルート / サブディレクトリ | 常時ロードされる前提知識・規約。インデックスに徹する |
| Rules | `.claude/rules/` | 制約・規約。`paths:`でスコープして必要時だけ載せる |
| Skills | `.claude/skills/` | 手続き・ワークフロー。本文は呼び出し時のみロード |
| Subagents | `.claude/agents/` | 隔離・並列タスク。最終メッセージだけ親へ返る |
| Hooks | `settings.json` 等 | 決定論的自動化。「必ず/禁止」は指示でなくこれで強制 |
| Output styles | `.claude/output-styles/` | 大きな役割変更専用。デフォルト指示を失う危険 |
| settings / permission | `settings.json` / `settings.local.json` | 権限・環境変数。秘密情報と過剰許可に注意 |

> append-system-prompt（CLIの`--append-system-prompt`フラグ）もブログが挙げるステアリング手段だが、
> セッション起動時に渡す実行時フラグでコミットされるファイルではないため、本スキルの監査対象外とする。

## なぜレポートを残すのか

設定の見直しは一度で終わらず、修正→再監査を繰り返す。ファイルに状態（OK/要改善/改善余地/
該当なし）と根拠と修正提案が残っていれば、何が片付いて何が宿題かを後から追える。
会話に流すだけだと取りこぼす。

このスキルは**機械的なチェックリストの穴埋めではない**。各項目が「なぜ重要か」を理解した上で、
そのプロジェクトの作りに即して本当に問題があるかを判断する。設定から断定できないものは無理に
OK/NGを付けず、正直に「要手動確認」として人間に委ねる。

## 全体の流れ

```
設定を検出  →  checklist.md を読む  →  機構ごとに監査  →  レポート生成・保存  →  会話で要約
```

## Step 1: 設定を検出する

まず対象プロジェクトにどのステアリング機構が存在するかを洗い出す。**無い機構は「該当なし」で
よいが、レポートには必ず全機構を載せる**（使っていないこと自体が判断材料になる）。

検出の当たり所（プロジェクトルートとサブディレクトリ両方を見る）:

- `CLAUDE.md` / `CLAUDE.local.md`（ルート＋各サブディレクトリ）。行数も測る
- `.claude/rules/**/*.md`
- `.claude/skills/*/SKILL.md`
- `.claude/agents/*.md`
- `.claude/output-styles/*.md`
- `.claude/settings.json` / `.claude/settings.local.json`（hooks・permissions・env）
- `.mcp.json` / settings内の`mcpServers`
- `.gitignore`（settings.local等が追跡対象になっていないか確認するため）

`.claude/plugins/`配下や、グローバル設定（`~/.claude/`）は**対象外**。監査するのは「このプロジェクトが
リポジトリとして抱えている設定」に限る（チームで共有される設定が点検対象だから）。

## Step 2: チェック項目の詳細を読む

`references/checklist.md` に、全機構の各項目について「何を確認するか・なぜ重要か・どう検出するか・
自動判定できるか手動確認が要るか・典型的な修正提案」をまとめてある。**監査を始める前に必ず読む。**
量があるので、まず目次で全体像を掴み、機構を監査するたびに該当節を見返す。

## Step 3: 機構ごとに監査する

`references/checklist.md` の順に機構を1つずつ点検する。各項目について:

1. **根拠を集める** — ファイルを実際に読み、grepで該当箇所を探す。推測で済ませない。
   例えば「rulesがスコープされているか」はrulesファイルのfrontmatterを実際に見て`paths:`の有無を確認する。
2. **状態を判定する** — 下記4状態のいずれかを付ける。
3. **根拠と修正提案を記録する** — 要改善・改善余地には必ず理由を添え、問題箇所は `file:line` で示す
   （クリックで飛べて修正が速い）。そして**どう直すか**を1〜2行で具体的に書く（移行先の機構、
   分割の仕方、追記すべきfrontmatterなど）。修正の適用まではせず、提案に留めてユーザーの承認に委ねる。

### 状態の定義

| 記号 | 状態 | 使うとき |
|------|------|----------|
| ✅ | OK | 適切に使えている／問題が見つからないことを設定から確認できた |
| ❌ | 要改善 | 明確なアンチパターン。`file:line` と何が問題かを示し、修正提案を添える |
| ⚠️ | 改善余地 / 要手動確認 | 問題の可能性はあるが断定できない、または好みの範囲。理由を添える |
| ➖ | 該当なし | その機構を使っていない（使うべきなのに無い場合は ⚠️ にして提案する） |

## Step 4: レポートを生成・保存する

監査結果を `claude-config-check-report.md`（対象プロジェクトのルート）に保存する。
既存があれば上書き前に日付サフィックスで退避するか、ユーザーに確認する。

レポート構成（この順序を守る）:

```markdown
# Claude Code設定 監査レポート

対象: <プロジェクトパス>  /  日付: <YYYY-MM-DD>

## サマリ
- 検出した機構: <CLAUDE.md / rules / skills / ... のうち存在したもの>
- 総評: <1〜3行。最も効いている点と、最優先で直すべき点>
- 件数: ✅ N / ❌ N / ⚠️ N / ➖ N

## 優先度つき修正提案
最優先（❌）から順に、項目・問題・修正提案を箇条書き。`file:line` を明記する。

## 機構ごとの詳細
### CLAUDE.md
| 項目 | 状態 | 根拠 (file:line) | 修正提案 |
| ... |
### Rules
### Skills
### Subagents
### Hooks
### Output styles
### settings / permission・セキュリティ
### 保守性・一貫性
```

レポート保存後、**会話には要約だけを返す**: 総評・件数・最優先の修正提案トップ3〜5件・
レポートの保存先パス。詳細はレポートを見てもらう。

## 監査の心得（誤検知を避ける）

- **小さく健全な設定を罰しない。** 50行のCLAUDE.mdは肥大化ではない。ブログが明示する目安は
  CLAUDE.mdの~200行のみで、これも「超えたら分割を検討」のラインに過ぎない。超えていなければ✅でよい。
- **存在しないことは必ずしも問題ではない。** hooksやsubagentsを使っていないプロジェクトは普通にある。
  「使うべき兆候（CLAUDE.mdに"必ず毎回"系の指示が何度も出てくる等）があるのに無い」ときだけ提案する。
- **断定できないなら ⚠️。** permissionが「過剰」かはプロジェクトの性質による。`Bash(*)`全許可のように
  明確なものは❌、グレーは⚠️で理由を添えて人間に委ねる。
- **秘密情報の検出だけは厳しめに。** 設定ファイルやhookスクリプトにAPIキー・トークンらしき文字列が
  直書きされていたら、確証が薄くても⚠️以上で必ず指摘する（漏洩は取り返しがつかない）。
